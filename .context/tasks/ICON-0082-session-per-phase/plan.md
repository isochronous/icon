## Task: ICON-0082
## Branch: feature/ICON-0082-session-per-phase
## Objective: Enable a workflow where each task-plan PHASE runs in its own fresh session (rather than one session spanning the whole task), launched by external automation (cron/trigger — user's chosen mechanism, accepting the harness-specificity/portability tradeoff). This requires HARDENING `plan.md` (and the `task-plan`/`phase-*` templates) so a cold, externally-launched run can resume any phase with sufficient hand-off information — no reliance on conversation continuity. Part 2 of the three-part pipeline (ICON-0081 graph ✓ merged → ICON-0082 session-per-phase → ICON-0083 terseness).
## Folder: .context/tasks/ICON-0082-session-per-phase/

## Decisions
- **Mechanism: external automation launches each phase** (user pick from the opening scoping question) — cron/trigger runs phase N, writes plan.md; a later run reads plan.md and runs phase N+1. Accepted tradeoff: least portable / most infra. EXACT trigger mechanism (cron via CronCreate, RemoteTrigger, a launcher command/script, or a documented external contract) is TBD — to clarify with user after exploring the current phase/handoff model.
- **Core necessity**: `plan.md` becomes the SOLE cold-start carrier between phase runs. The `task-plan` skill already frames plan.md as "enough context for a different person/agent on a different machine to resume cold" — this task HARDENS that from aspiration to guarantee for the per-phase-session case.
- Architecture-primary: mechanism + handoff-contract design via @architect before implementation (mirrors ICON-0081's design-first approach).
- **USER SCOPING DECISION**: rather than ICON shipping a fixed harness-specific launcher (which would pin portability / breach ADR-004), ICON ships a **launcher-GENERATOR skill** that emits a per-phase launcher script parameterized by a **`target-harness` config option** (claude-code / copilot-cli / generic cron/CI). The portable core (plan.md hardening + phase-reconstruct-on-entry + formalized `{task_id, task_folder, phase}` entrypoint) ships as tool-agnostic instruction-content; the harness-specific launcher is GENERATED on demand for the consumer's chosen harness. This keeps ICON itself portable (a skill that produces the launcher) while delivering working automation — the ICON-idiomatic resolution (ICON ships skills that produce artifacts, like the `initialize-*` / `context-specialist-*` generators).
