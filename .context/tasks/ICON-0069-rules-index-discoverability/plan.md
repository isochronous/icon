## Refinement: script-backed freshness (user-directed — reduce non-determinism + token usage)
**Concern**: Layer 1 (consumer task-close check) was model-enforced — the model hand-scans 3 dirs vs the index every close (tokens + drift risk). And the hook (Layer 2) has the SAME completeness logic inlined → two implementations that can drift.
**Investigation (Explore)**: ONE shared script is feasible — `skills/context-maintenance/scripts/check-rules-index.sh`. ICON hook calls it via `$repo_root/skills/context-maintenance/scripts/check-rules-index.sh` (in ICON, `$repo_root` IS the plugin root); consumer maintenance pass calls the SAME physical file via `${CLAUDE_SKILL_DIR}/scripts/check-rules-index.sh` (Claude Code) / Copilot install path. Both reach one copy.
- **No 3-copy parity burden**: that's specific to scripts invoked by MULTIPLE skills (append-retrospective-entry × 3). This is invoked by ONE skill + the hook (hook is exempt from the "skills can't share scripts" rule) → ONE copy.
- **`.sh`-only**: `shell-portability.md` does NOT mandate `.ps1`; the hook is bash-only; the model invokes via bash on all Claude Code platforms. (`.ps1` deferrable if Windows-without-WSL consumers ever need it.)
- **New pattern**: hook calling an external script has zero precedent in `.githooks/pre-commit` (all inline today) — acceptable (it already shells out to diff/grep/awk); script must fail-safe if missing.
**Decision (Option B — unify)**: extract the completeness logic into `check-rules-index.sh` as SSOT; the hook REPLACES its inline `rules_index_check_needed` block with a call to it; `context-maintenance` replaces the model-enforced scan with a "run the script" instruction (keep the audit-row as the human-readable description, mechanism = script). ## Gap fix: split-vs-freshness interaction (user-identified — full fix, MR !53)
**Gap**: the canonical split procedure (`context-document-guidelines` + `context-maintenance` File Size Rule, ≥16KB + ≥3 `##`) does `git rm <name>.md` and creates `<name>/README.md`. After a by-the-book split of an indexed file: the old `(standards/<name>.md)` row dead-links (NOT caught — dead-ref hook check skips `.context/`), and the new `<name>/README.md` is sub-dir-invisible to the freshness check → content silently drops from the index. The `skill-decomposition.md` precedent the check was modeled on *deviated* from the spec (kept the top-level index file). `workflows/task-plan/` is already a pure dir whose row nothing enforces.
**Decision (user: full fix)**:
1. **Harden `check-rules-index.sh` — folder-aware + bidirectional**: indexable unit = each top-level NAME under `standards/`/`workflows/` (a `*.md` file OR a sub-dir; collapse file+dir pairs to one name) → require ≥1 row whose link target is `(<dir>/<name>.md)` or `(<dir>/<name>/...)`; ADRs unchanged. PLUS backward check: every `(standards|workflows|decisions/...)` row link must point at an existing path (catches dead rows). Avoid substring false-positives (boundary after name = `.md)` or `/`).
2. **Hook trigger must also fire on sub-dir changes** under standards/workflows (today it SKIPS `*/*`) — else a pure new split-folder never triggers the check (closes scenario c).
3. **Make the split procedure rules-index-aware**: a step in BOTH the `context-document-guidelines` Folder Split Rule and the `context-maintenance` Phase-3 File Size Rule — "if the split file had a `rules-index.md` row, repoint it at the new `<name>/` folder, same change."
