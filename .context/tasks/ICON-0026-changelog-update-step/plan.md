## Task: ICON-0026
## Branch: feature/ICON-0026-changelog-update-step
## Objective: Add a `[Unreleased]` CHANGELOG update step to the local task-completion phase template, and ship a new repo-level skill that codifies how to write a good entry (terse-for-internal vs efficient-for-user-facing, plus the cumulative-effect rule for overriding prior unreleased entries).
## Folder: .context/tasks/ICON-0026-changelog-update-step/

## User Directive (verbatim)

> "Add a step to the task completion template for this repo (the one in ./.context, not the one in context_template) to update the changelog's 'unreleased' section with a line item. Let's also add a repo-level skill for writing a good changelog message: Be terse for internal-only changes. For user-facing changes, use efficient language to describe both what the change is and how it affects them. If a change updates over overrides another unreleased feature that has been documented in the changelog, only the end result should be included. For example, if commit 1 was 'changed const foo from 3 to 5', and commit 2 was 'changed const foo from 5 to 3', there should be no changelog entry. If commit 2 was instead 'changed const foo from 5 to 8' then there should be a single changelog entry 'changed const foo from 3 to 8'."

Note: The "local override, not the template mirror" framing is intentional — this is a repo-local convention for ICON's own dev workflow (the plugin maintainer authors `[Unreleased]` entries as features land). Consumer repos generated by `initialize-repo` get the unmodified base template; this convention is specific to repos with a versioned CHANGELOG and the `release-plugin` flow. **Skip the three-surface rule on purpose here.**

## Decisions

- **Step lives in `.context/workflows/task-plan/phase-completion.md` only.** Per user explicit direction, do NOT mirror into `context_template/context/workflows/task-plan/phase-completion.md`, and do NOT update `skills/task-plan-phase-completion/SKILL.md` (the shipped fallback). The three-surface rule from ICON-0014 is deliberately bypassed here because the changelog convention is ICON-repo-specific.
- **New skill location: `.claude/skills/changelog-entry/`.** Mirrors the `.claude/skills/release-plugin/` precedent for repo-level (maintainer-only, not shipped to consumers) skills. `user-invocable: true` so the maintainer can invoke it directly at task close.
- **Skill is complementary to `.context/standards/changelog-discipline.md`.** The standards file documents *what makes a good entry* (one sentence, no block formatting, ticket IDs at end). The new skill documents *the procedure for writing one at task close* (when to invoke, how to decide internal-vs-user-facing tone, the cumulative-effect rule for overriding prior unreleased entries). Cross-reference; don't duplicate.
- **Where to insert the new step in `phase-completion.md`.** The Context Update Checklist already covers `.context/` updates. The CHANGELOG update is task-output-facing, not `.context/`-facing, so it belongs as a distinct step — recommend a new `## Update CHANGELOG [Unreleased]` section between the @reviewer Delegation Template and the Context Update Checklist (or just after reconcile-plan.md). @coder picks the exact placement based on flow.
