## Task: ICON-0012
## Branch: feature/ICON-0012-plugin-scoped-manager-default-hook
## Objective: Move the manager-role SessionStart hook from user-scope `~/.claude/settings.json` into the plugin's own `hooks/hooks.json` so `${CLAUDE_PLUGIN_ROOT}` resolves correctly and the hook follows plugin upgrades automatically. Replace the user-settings opt-in/out flow with a generic `~/.claude/icon-user-settings.json` config file, reusable for future ICON user-level settings.
## Folder: .context/tasks/ICON-0012-plugin-scoped-manager-default-hook/

## Decisions
- **Hook location → plugin-scoped `hooks/hooks.json`**: Claude Code only resolves `${CLAUDE_PLUGIN_ROOT}` inside a plugin's own `hooks/hooks.json`, not in user-scope `settings.json`. The current `/ICON:enable-manager-default` skill writes a `${CLAUDE_PLUGIN_ROOT}`-bearing entry into user settings, which fails at session-start with "Hook command references ${CLAUDE_PLUGIN_ROOT} but the hook is not associated with a plugin." Moving the wiring into the plugin fixes the variable scope, ties the hook to the active plugin version automatically, and removes the need for user-side install steps.
- **Opt-out via `~/.claude/icon-user-settings.json`**: Replaces per-user `enable`/`disable` slash commands that edited the SessionStart hook list. New schema: a single JSON file at `~/.claude/icon-user-settings.json` holding a top-level object — `managerDefault` is the only key for this task; future ICON user-level settings extend the same file. Default behavior when file is absent or `managerDefault` is unset: inject manager role (manager-default is on). To opt out, the user writes `{ "managerDefault": false }` (via the rewritten `/ICON:disable-manager-default` command).
- **Migration of legacy user-settings entries**: The rewritten `enable`/`disable` commands also detect and remove any legacy `inject-manager-role` entry from `~/.claude/settings.json`. Without this, users who installed the previous version retain a broken entry (which produces the very `${CLAUDE_PLUGIN_ROOT}` error this task fixes). Migration is idempotent — re-running either command is safe.
- **Cross-platform via single Node.js wrapper**: Claude Code's plugin `hooks/hooks.json` has no documented per-platform conditional, and listing both bash and pwsh commands causes both to run on dual-shell hosts (not a fallback). The only viable cross-platform shape per official docs is a single command that resolves identically on every host. We consolidate the two existing scripts (`inject-manager-role.sh`, `inject-manager-role.ps1`) into one `hooks/inject-manager-role.mjs`. Node.js is a safe assumption because Claude Code itself is a Node.js CLI. Research cached at `.context/cache/claude-code-plugin-hooks-2026-05-20.md`.
- **Delete legacy bash + pwsh scripts**: Keeping them alongside the `.mjs` creates two ambiguous code paths and revives the script-parity bugs prior retrospectives warn about. The `.mjs` becomes the single source of truth; the `.sh` and `.ps1` files are removed in the same commit. Anyone with a legacy user-settings hook pointing at them gets cleaned up by the rewritten `enable`/`disable` commands.
- **Hook command shape**: `{ "type": "command", "command": "node", "args": ["${CLAUDE_PLUGIN_ROOT}/hooks/inject-manager-role.mjs"] }` — exec form per Anthropic guidance, with `${CLAUDE_PLUGIN_ROOT}` resolving inside the plugin scope. SessionStart matcher group covers both `startup` and `resume` so `/clear` re-injection continues to work.

## Key Files
- `hooks/hooks.json` *(NEW)*: Wires `SessionStart` (matcher `startup|resume`) → `node "${CLAUDE_PLUGIN_ROOT}/hooks/inject-manager-role.mjs"`. The single source of truth for hook wiring in v1.16+.
- `hooks/inject-manager-role.mjs` *(NEW)*: Cross-platform Node.js port of the existing bash + pwsh logic. Steps in order: (1) exit 0 if `CLAUDE_PROJECT_DIR` unset or `${CLAUDE_PROJECT_DIR}/.context` absent; (2) read `${HOME}/.claude/icon-user-settings.json` (or `%USERPROFILE%\.claude\icon-user-settings.json`); if it parses and `managerDefault === false`, exit 0 silently; if it fails to parse, emit a stderr warning and continue (fail-open); (3) require `CLAUDE_PLUGIN_ROOT` to be set, else stderr warn + exit 0; (4) require `${CLAUDE_PLUGIN_ROOT}/agents/manager.agent.md` to exist, else stderr warn + exit 0; (5) emit the JSON envelope (`hookSpecificOutput.additionalContext`) with the same discipline-led prefix + manager body the current scripts produce. Use `JSON.stringify` for escaping.
- `hooks/inject-manager-role.sh` *(DELETE)*: Removed in favor of the `.mjs` wrapper. The `inject-manager-role` substring stays alive in the `.mjs` filename so existing user-settings cleanup scripts (which substring-match this token) continue to identify legacy entries.
- `hooks/inject-manager-role.ps1` *(DELETE)*: Removed alongside the bash variant.
- `commands/enable-manager-default.md` *(REWRITE)*: New behavior — (1) clean up any legacy `inject-manager-role` entry from `~/.claude/settings.json`; (2) write `{ "managerDefault": true }` to `~/.claude/icon-user-settings.json`, preserving any other top-level keys; (3) report the migration outcome and current state. Drop the bash/pwsh probe and the `${CLAUDE_PLUGIN_ROOT}` snippet — no longer relevant.
- `commands/disable-manager-default.md` *(REWRITE)*: New behavior — (1) clean up any legacy `inject-manager-role` entry from `~/.claude/settings.json`; (2) write `{ "managerDefault": false }` to `~/.claude/icon-user-settings.json`; (3) report. Same drop of legacy logic.
- `CHANGELOG.md` *(UPDATE)*: Entry under next unreleased section documenting the migration, the new file location, and what existing users should expect (legacy entries auto-cleaned on next `/ICON:enable-manager-default` or `/ICON:disable-manager-default`).
- `.claude-plugin/plugin.json` *(version bump)*: Handled by `release-plugin` skill at task close — not in this task's direct scope.
- `.context/retrospectives.md` *(append)*: Manager-written entry summarizing the migration and lessons learned about `${CLAUDE_PLUGIN_ROOT}` scoping.
- `.context/domains/hooks.md` *(audit; create or update)*: If a domain file for hooks doesn't already cover plugin-scoped hook wiring vs user-settings hooks, `@context-specialist` should create or extend one during retrospective.

## Constraints
- ICON is pure-content (no compile/test/package manager) — see ADR-005. Verification is by inspection + manual run of the hook scripts; no test suite to gate this work.
- `.claude-plugin/plugin.json` is the version SSOT — see ADR-003. Version bump happens at release time via `release-plugin`, not within this task.
- **Backward compatibility**: Users on the previous plugin version have a `${CLAUDE_PLUGIN_ROOT}`-bearing entry in their `~/.claude/settings.json` that produces an error every session. The new `enable`/`disable` commands MUST clean that up automatically — surfacing instructions to manually edit user settings is not acceptable for a maintenance workflow.
- **Cross-platform**: Bash variant covers Linux, macOS, WSL, Git-Bash. PowerShell variant covers pure-Windows hosts without bash. Both shells' opt-out logic must be equivalent — divergence between `.sh` and `.ps1` has bitten this repo before (see prior retrospectives on hook parity).
