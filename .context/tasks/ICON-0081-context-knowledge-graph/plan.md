## Task: ICON-0081
## Branch: feature/ICON-0081-context-knowledge-graph
## Objective: Extend ICON's context read/write process so that an in-memory, on-demand knowledge graph — built by parsing `.context/` — is constructed and traversed during BOTH context discovery (for full discoverability of transitively-related context) and context update/maintenance (to keep `.context/` consistent and current: catch dangling `[[links]]`, orphaned files, missing cross-references, and drift). This is Part 1 of a three-part pipeline (ICON-0081 KG → ICON-0082 session-per-phase → ICON-0083 terseness).
## Folder: .context/tasks/ICON-0081-context-knowledge-graph/

## Decisions
- **In-memory, on-demand graph (not a persisted artifact, not an external/MCP server)**: User directive — "for repos of the size I expect to work on, a simple in-memory graph populated on-demand should be sufficient." Built in-context by parsing `.context/` when needed; no new committed data file, no external dependency. Honors ICON portability (works on Claude Code + Copilot CLI) and ADR-005 (pure-content, no runtime).
- **Two-sided use**: The graph serves BOTH the read side (context discovery → traverse relations so no related context is missed) AND the write side (context update/maintenance → use the graph to keep `.context/` up-to-date and internally consistent). User clarification: "that knowledge graph should be then used during both context discovery and the context update step to ensure both full discoverability and that .context stays up-to-date."
- Graph model (entity types, relation types, build procedure, traversal procedure) to be designed by @architect via `design-first` before any @coder work — this is architecture-primary.
- **USER SIGN-OFF (2026-07-17) — chose the fuller build**: (1) Mechanism 3 script **+ format seams now** (NOT deferred) — augment the `.context/` format with first-class edge metadata; accept `context_template/` ripple + `iconrc.json` version bump + consumer `/upgrade-repo` propagation. (2) Exclude `tasks/*/plan.md` from the graph (recommended). (3) `--check` **also becomes a fail-closed `.githooks/pre-commit` gate** (NOT maintenance-only). → Design must be EXTENDED (@architect continued) to spec the format-seam concretely (what metadata, which node types, how impl-* generators emit it, template mirror) and the pre-commit gate (fail-closed design avoiding ICON-0075 grep leading-dash / fail-open traps; interplay with existing rules-index + dead-ref gates; false-positive management). ADR-012 promotion confirmed.
- **@architect v1 recommendation (design.md, SUPERSEDED by user sign-off above on knobs 1 & 3)**: Mechanism 3 (script-materialized) + thin procedural layer; reject Mechanism 2 (format-augmented) for v1. Rationale: the discoverability/consistency edges ALREADY exist as parseable signals (rules-index rows, prose `[text](path)` links, ADR "Superseded by", folder README tables, retrospective "Promoted to:", `excludes`) — a deterministic parser extracts them with no format change, beating both the "agent mentally builds a graph" reliability gap (Mech 1) and the high-ripple format change that forces the `iconrc` bump + ships to every consumer (Mech 2). Graph model: closed node set (existing `.context/` doc types) + closed 6-edge set (`references`, `indexed-by`, `supersedes`/`superseded-by`, `covers`, `promoted-from`, `excludes`). New script `skills/context-maintenance/scripts/context-graph.{sh,ps1}`: `--emit` (read side, compact adjacency list, no content — agent does bounded BFS from rules-index/task seeds, gated by existing simple-vs-complex heuristic so ADR-008 budget is respected) + `--check` (write side, generalizes `check-rules-index.sh` from 3 dirs to whole tree: dangling refs across all prose links, orphan/unreachable nodes, missing-cross-ref candidates flagged). Leaves `check-rules-index.sh` and the pre-commit dead-ref resolver untouched; `--check` is maintenance-only (not a new pre-commit gate) for v1. File set: NEW `context-graph.{sh,ps1}`; MODIFIED `agents/manager.agent.md`, `skills/context-maintenance/SKILL.md`, `README.md` (script registration). NO `context_template/` touch → no iconrc bump. Recommends promoting the decision to **ADR-012** (+ its own rules-index row). One honest gap deferred: by-name `skill § Section` prose refs aren't machine-parseable (the only real format-seam candidate) — deferred as evidence-gated follow-up.
- **Expanded latitude (user directive)**: "If we need to adjust the format of context to include specific seams/edges, or create a format script for querying .context at task start, that's fine." So @architect may consider three (composable) mechanisms, not just the conceptual one: (1) purely procedural/in-context graph from existing signals (`[[links]]`, rules-index rows, ADR supersedes, domain coverage); (2) **format-augmented** — add explicit seams/edges to the `.context/` document format (e.g. structured frontmatter / relations block) so edges are first-class; (3) **script-materialized** — a query/format script that parses `.context/` at task start and emits nodes/edges (the "in-memory graph, populated on-demand"). (2)+(3) compose naturally. Weigh reliability gain against ripple: (2) touches `context-document-guidelines`, the `context-specialist-*` builders, `context_template/`, and potentially every existing `.context/` file; (3) adds a script consistent with existing `hooks/*.mjs` + `skills/*/scripts/*.sh` infra under ADR-005.
