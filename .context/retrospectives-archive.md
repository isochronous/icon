### ICON-0069: Rules-index discoverability (standards/workflows/decisions)
- **Avoid**: A change that adds OR modifies any file under `context_template/` must bump the template version in `context_template/context/iconrc.json` in the same commit — the ICON-0044 pre-commit gate enforces it (that version is what `/upgrade-repo` Phase 2 uses to decide whether to push template updates to consumers; without the bump, consumer repos silently miss the new content). This is the TEMPLATE version, distinct from the plugin release version (`.claude-plugin/plugin.json`, ADR-003), which stays untouched mid-task. The @architect spec's "Key Files" list omitted this coupling and the @coder hit the gate only at staging time, costing a follow-up round. When scoping any task that touches `context_template/`, list the iconrc template-version bump as a required companion edit up front.
- **Repeat**: The infrastructure-change playbook held again (after ICON-0065/0067/0068): read-only Explore enumerate → @architect exact-edit-spec (per-site strings + grep-checkable acceptance + "do NOT touch" list + multi-copy sync set + an UP-FRONT ADR-008 word-count of the always-loaded delta) → @coder applies → @reviewer checkpoint DURING implementation. Three design calls that paid off: (1) parent-row granularity for the index (one row per indexed sub-dir — the index ROUTES, it does not mirror the tree) kept it terse and churn-resistant; (2) fail-safe gate wording ("an absent index is not a blocker") so the change degrades gracefully in consumer repos that haven't regenerated; (3) matching mechanism to artifact nature — generate-by-scanning for the derived index + create-if-absent on upgrade (NOT the version-marker logic used for customizable templates), because the index derives from the repo's own rule files and is always safe to regenerate.
- **Repeat**: When shipping a hand-curated or generated index/table, ship its freshness mechanism in the SAME change — don't accept "it'll drift" as a trade-off (the @reviewer flagged rules-index drift as an accepted minor; the user correctly overrode it). The proportionate pattern: a consumer-reaching soft check at task-close (a `context-maintenance` audit-table "Index-coverage gap" row) PLUS a hard pre-commit completeness gate for the maintainer repo, mirroring the existing O-V1 skill-registration directory-vs-index check. Note the asymmetry that drove the design: the pre-commit hook is ICON-only (does NOT ship to consumers), so the consumer-drift fix HAD to live in the consumer-shipped `context-maintenance` skill, not the hook.
- **Repeat**: Prefer a single shared SCRIPT as the source of truth for a mechanical invariant over (a) model-enforced hand-scans and (b) logic duplicated across call sites. Here the rules-index completeness check started as model-enforced in `context-maintenance` (token cost + non-determinism every task close) plus the SAME logic inlined in the pre-commit hook (drift risk). Extracting `skills/context-maintenance/scripts/check-rules-index.sh` and having BOTH the hook and the consumer task-close audit call it made the check deterministic, near-zero-token, and single-source. Enabler: a skill-resident script is reachable by the ICON hook (`$repo_root/skills/.../scripts/`) AND consumers (`${CLAUDE_SKILL_DIR}/scripts/`) — same physical file, one copy (the 3-copy parity burden only applies to scripts invoked by multiple skills). When a check is mechanical and runs in a model loop, reach for a script the model RUNS, not prose it executes by hand.
- **Avoid**: A new gate that asserts a structural invariant must be validated against the repo's OTHER structural operations before shipping — and modeled on the WRITTEN contract, not a convenient existing example. The rules-index freshness check assumed "parent-row granularity" (sub-dir files covered by a top-level index file's row), modeled on `skill-decomposition.md`; but that precedent had DEVIATED from the canonical file-split spec, which `git rm`s the top-level file and creates `<name>/README.md` — so a by-the-book split silently dropped content from the index (dead row uncaught; new folder sub-dir-invisible). The fix that generalizes: make a completeness gate BIDIRECTIONAL (forward — every unit has a row; backward — every row resolves on disk) and FOLDER-AWARE (a unit is a name = file OR sub-dir), and add the index-update obligation at the mutating operation's own site (the split procedure), not only in a separate maintenance rule. When adding a gate keyed on file structure, enumerate what moves/renames/splits/deletes files in this repo and test the gate against each.
