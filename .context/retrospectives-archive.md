### ICON-0073: Harness-enforced guardrails (#40) — verify external-tool behavior by live execution, not docs; fail-open or brick the session
- **Avoid**: A spec built from official docs + the tool's own SDK type definitions was wrong on three counts that only live execution against the actual harness exposed — and one would have bricked every session. For the Copilot CLI PreToolUse hook, the doc/SDK-derived design assumed (a) a separate `.github/hooks/guardrail.json` was needed (wrong — Copilot's `loadPluginHooks` reads the plugin's own `hooks/hooks.json` directly, and ICON ships as a plugin so writing into a consumer's `.github/` would be wrong anyway); (b) hook stdin is camelCase `toolName`/`toolArgs` per the SDK `PreToolUseHookInput` type (wrong — both harnesses deliver snake_case `tool_name`/`tool_input` to the external command hook); (c) the Claude-style `command`+`args` entry form executes on Copilot (wrong — it is inert in Copilot's executor and fail-CLOSES with "code 1"; only the `bash` shell-string field runs). Because a PreToolUse hook that errors makes Copilot fail-CLOSED and deny EVERY tool call, shipping the unverified design would have bricked every Copilot session on install. A live probe (patch the installed plugin's `hooks/hooks.json`, run `copilot -p`, capture stdin/env/exit, restore) caught all three before any commit. When a feature integrates with an external tool/harness, verify its runtime contract by executing against the real tool — do not trust documentation or SDK type definitions, especially for a fail-closed enforcement path.
- **Repeat**: (1) The fail-OPEN invariant is non-negotiable for an enforcement hook on a fail-closed harness: the script wraps its entire body and ALWAYS `process.exit(0)`, emitting a deny only on a positive rule match; empty/malformed/unknown stdin and any thrown error → allow. The reviewer adversarially tested this (garbage, null, array, missing fields, regex-special tool names) and it held. Design enforcement hooks so the WORST case is "control momentarily not enforced" (with prose as backstop), never "all tools denied". (2) Single-artifact portability over per-harness duplication: ONE `hooks/hooks.json` entry carrying BOTH the `command`+`args` (Claude) and `bash` (Copilot) field forms, plus ONE script that selects output shape by `transcript_path` presence, serves both harnesses — verified live on Copilot 1.0.63 (benign allowed, `wget|sh` denied end-to-end with the agent then adapting to download→inspect→run) and by fixtures for the Claude nested-output shape. (3) Recon → research → @architect exact-edit-spec → @coder → @reviewer held again, but with a decisive added step: an empirical-verification gate between spec and implementation whenever the spec rests on external-tool behavior. ### ICON-0075: Security CI pipeline + pre-commit gates (#42) — ground CI in real infra, coordinate shared-file ownership, and the grep leading-dash fail-open trap
- **Avoid (grep leading-dash → silent fail-OPEN)**: The secret-scan spec used `grep -Eq "$re"`; the `pem-private-key` pattern starts with `-----`, which grep parsed as options (exit 2, "unrecognized option") on every file. Because the call was guarded by `if printf … | grep -Eq …`, the `if` treats grep's exit-2 error as "no match" — so the PEM rule would have silently NEVER fired: a fail-OPEN security gate that looks like it works. The @coder caught it and fixed with `grep -Eq -e "$re"` (explicit `-e` end-of-options). Whenever a regex/string passed to grep can begin with `-`, use `grep -e <pat>` (or `--`); and remember an `if grep` guard masks grep's own errors, so a malformed pattern fails silently-open — test each pattern against a known-positive fixture rather than trusting that "no error" means "working".
- **Repeat (ground CI in real infra + coordinate shared-file ownership)**: Before designing the pipeline, inspected a past pipeline's jobs to learn the ACTUAL runner — a project runner tagged `devqa-common-saas` (Docker executor), NOT GitLab.com shared runners — so every job must carry that tag or it silently never runs. Before adding/designing a root infra file (CI config, etc.): (a) ground the design in the project's real runner/infra by inspecting existing pipeline jobs, and (b) check for in-flight branches/MRs that own the same file and surface the coordination decision to the user — never unilaterally clobber a colleague's pending work.
